<li>限流详解

<ol>
<li>限流算法

<ol>
<li>令牌法：每秒产生指定数量令牌，如果能够获得令牌则不限流；无法获得令牌则限流</li>
<li>漏斗法：类似于一边进水一边放水；将请求放入桶中如果桶没满，正常处理，如果桶满，则拒绝请求。</li>
</ol></li>
<li>应用级限流

<ol>
<li></li>
</ol></li>

<ol>
<li>redis + lua</li>
<li>nginx + lua</li>
</ol></li>
<li>接入层限流 (nginx等)

<ol>
<li>ngx_http_limit_conn_module   limit_conn  对某个key对应的总的网络连接数进行限制
示例：
<code>// 按域名限流
limit_conn_zone $serve_name zone=one:10m rate=1r/s;
location ~* .htm$ {
limit_conn zone=one burst=5  nodelay;
proxy_pass http://backend_tomcat;
}</code></li>
<li>ngx_http_limit_req_module   limit_req  对某个key对应的请求进行限流，是使用漏斗算法实现的
<code>// 按ip限流
limit_req_zone $binary_remote_addr zone=one:10m rate=1r/s;
location ~* .htm$ {
limit_req zone=one burst=5  nodelay;
proxy_pass http://backend_tomcat;
}</code>
ngx_http_limit_conn_module，ngx_http_limit_req_module 两个模块都可以针对Ip或域名进行限流</li>
<li>lua-resty-limit-traffic  openResty提供的lua限流模块
可以灵活编程限流</li>
</ol></li>
<li>节流

在特定的时间窗口内对重复的相同的时间最多只处理一次，或者想限制多个连续相同的时间最小执行时间间隔

<ol>
<li>三种实现方式 throttleFirst throttleLast throttleWithTimeout</li>
<li>throttleFirst/throttleLast是指在一个时间窗口期内如果有多个相同的请求只处理第一个/最后一个。比如连续拖拽浏览器窗口，只响应最后一次</li>
<li>throttleWithTimeout限制两个连续事件的执行时间不得小于某个时间窗口。比如搜索关键字自动补全，利用该方式可以减少网络请求</li>
</ol></li>
</ol></li>
<li>降级

<ol>
<li>降级预案 分级降级

<ol>
<li>一般：因网络抖动或项目上线造成的超时，可自动降级</li>
<li>警告：一段时间内成功率有波动（95%~100%），可自动降级，并发送告警</li>
<li>错误：可用率低于90%，或访问量突增达到系统阈值，要根据情况实现自动降级或人工降级</li>
<li>严重错误：由于特殊原因出现的错误，需要实现紧急人工降级</li>
</ol></li>

<li>回滚机制

<blockquote>
  回滚是指当程序或数据出错时，将程序或数据恢复到最近一个正确的版本，常见的如事物回滚，代码库回滚，部署版本回滚，数据库版本回滚，静态资源版本回滚。通过回滚机制可以保证系统在某些场景下的高可用。
</blockquote>

<li>分类

<li>多级缓存： 

<ol>
<li>本地缓存过期时间设置为远程缓存的一半，防止本地缓存时间过长造成多实例间数据不一致</li>
<li>写缓存的时候，先写本地缓存，然后异步更新远程缓存；异步更新远程缓存是为了更快的相应用户请求。</li>
<li>读缓存时，先读本地缓存，本地缓存不命中再批量查询远程缓存</li>
</ol></li>
<li>缓存使用模式实践

<ol>
<li>主要分为两大类：

<ol>
<li>Cache-Aside 由业务代码直接维护缓存</li>
<li>Cache-As-SoR(read-through（读穿透）,write-through（写穿透）,write-behind（回写，即异步写）)： 业务代码只操作缓存，然后委托给缓存去操作SoR

<ol>
<li>read-through: 业务代码调用cache，如果未命中由cache回源到SoR</li>
<li>write-through:业务代码调用cache写，然后由cache负责写缓存和SoR，同步写cache和SoR</li>
<li>write-behind:业务代码调用cache写，然后由cache负责写缓存和SoR,异步写cache和SoR，异步后可以实现批量写、合并写、延时、限流</li>
</ol></li>
</ol></li>
<li>名词解释

<ol>
<li>SoR 即system of record 记录系统或者叫做数据源，即实际存储原始数据的系统</li>
<li>Cache：缓存，SoR的快照 cache的访问速度要比SoR要快</li>
<li>回源：即到数据源头查找数据</li>
</ol></li>
</ol></li>
</ol></li>

<li><p>如何实现扩容

<ol>
<li>扩容的三个步骤

<ol>
<li>尝试简单扩容解决</li>
<li>水平拆分和垂直拆分数据库/应用系统的伸缩性，即通过扩容提升系统负载能力</li>
<li>根据现有系统特性，从架构层面进行重构甚至重新设计</li>
</ol></li>
<li>单体应用垂直扩容

<ol>
<li>增大机器内存，增加核数，换固态硬盘，增加磁盘容量</li>
</ol></li>
<li>单体应用水平扩容

<ol>
<li>增加镜像，利用负载均衡器负载新增加的应用</li>
</ol></li>
<li>应用拆分：按业务将一个大系统拆分为多个子系统</li>
<li>数据库拆分

<ol>
<li>跨库join可以考虑通过全局表、ES搜索等异构数据机制来实现</li>
<li>还要考虑分布式事务</li>
</ol></li>
<li>扩容总结：应用扩容可以通过部署更多的实例来解决，无法部署更多实例时，就需要考虑拆分或者重新架构。而数据库扩容首先是硬件层面的，然后按照业务垂直拆分，接着水平拆分，最后根据场景进行读写分离，还可以将读流量分流到NoSql中</li>
