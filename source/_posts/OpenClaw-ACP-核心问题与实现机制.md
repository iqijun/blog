## 核心问题
1. **工具碎片化与集成成本高昂**  
   - 每个代码编辑器或 IDE 需要为不同 AI 代理编写定制集成代码  
   - 导致高集成成本、有限兼容性和厂商锁定

2. **上下文缺失与工作流中断**  
   - AI 编程助手在会话间无法维持上下文  
   - 开发者需要重复提供背景信息

3. **平台能力受限**  
   - OpenClaw 原生子代理受沙盒安全和工具集限制  
   - 专业编码工具（如 Claude Code）孤立运行

4. **专业 AI 代理难以协作**  
   - 难以集成到团队协作平台（如 Slack/Discord）  
   - 无法利用 OpenClaw 的路由和记忆管理能力

5. **通用协议对 AI 支持不足**  
   - REST/gRPC 难以支持服务端主动推送、流式输出  
   - 缺乏内置幂等性和会话上下文支持

6. **多代理调试困难**  
   - 复杂工作流中难以追踪问题根源

## 核心机制
1. **标准化通信协议**  
   - 基于 JSON-RPC 2.0 的双向事件协议

2. **桥接架构**  
   - `openclaw acp` 命令行工具作为 IDE 与 Gateway 的桥梁

3. **会话管理**  
   - 映射 ACP 会话到 Gateway 会话键  
   - 支持会话标签和隔离机制

4. **服务端事件推送**  
   - 主动推送 `assistant.text.delta` 等事件

5. **多渠道一致性**  
   - 标准化消息格式保证 Web/WhatsApp/CLI 体验一致

6. **外部代理集成**  
   - 通过 `acpx` 运行外部编码工具（如 Claude Code）

7. **溯源跟踪 ID**  
   - 元数据跟踪命令来源和路径

## 实现机制
- **命令行工具**：`openclaw acp` 通过 stdio 与 IDE 通信
- **WebSocket 通信**：转发提示到 Gateway
- **会话存储**：管理创建/加载/淘汰机制
- **策略守卫**：`policy.ts` 控制功能启用和访问权限
- **SDK 集成**：使用官方 `@agentclientprotocol/sdk`
